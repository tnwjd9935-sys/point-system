# Point System

무료 포인트의 **적립/적립취소/사용/사용취소** 흐름을 `거래 원장(POINT_TRADE)`과 `상세 이력(POINT_DETAIL)` 기반으로 관리하는 API입니다. 잔고는 `USER_BALANCE`에 현재 가용 금액만 유지합니다.

## 기술 스택

- Java 21
- Spring Boot 3.4.3
- Gradle
- H2
- JdbcTemplate 
- Lombok

## 실행 방법

```bash
# 실행
./gradlew bootRun
```

```bash
# 테스트
./gradlew test
```

## API 목록

| Method | Path | 설명 |
|---|---|---|
| POST | `/api/points/earn` | 적립(PO01) |
| POST | `/api/points/earn/cancel` | 적립취소(PO03) |
| POST | `/api/points/use` | 사용(PO02) |
| POST | `/api/points/use/cancel` | 사용취소(PO04) |

## 테이블 역할

- `USER_BALANCE`
  - 사용자별 **현재 가용 잔고**를 저장합니다.
  - 거래 원장 전체를 매번 합산하지 않고, 조회/검증에 필요한 값을 빠르게 가져오려는 목적입니다.

- `POINT_TRADE`
  - 포인트의 “거래 한 건”을 원장 형태로 저장합니다(적립/사용/취소/재적립).
  - `amount`는 거래 금액, `remain_amount`는 거래 유형에 따라 **남은 처리 가능 금액**으로 사용합니다.

- `POINT_DETAIL`
  - 어떤 적립 건이 어떤 사용/취소에 연결되었는지 **1원(1포인트) 단위로 추적**하기 위한 상세 이력입니다.
  - 사용(PO02) 시에는 어떤 적립 건에서 얼마를 차감했는지(`source_point_key`)를 남기고, 사용취소(PO04) 시에도 복원/재적립 흐름을 상세로 남깁니다.

## 상태/거래 코드

### 거래 타입(`POINT_TRADE.trade_type`)

- `PO01`: 적립
- `PO02`: 사용
- `PO03`: 적립취소
- `PO04`: 사용취소
- `PO05`: 재적립(사용취소 시 만료분 대체)

### 상태(`POINT_TRADE.status`)

- `PS01`: 유효
- `PS02`: 만료
- `PS03`: 취소(무효)
- `PS04`: 사용취소됨(원 PO02가 전액 취소된 상태)

## 핵심 설계

### 잔고와 원장을 분리한 이유

`USER_BALANCE`에는 “현재 가용 잔고”만 둡니다. 검증(가용 잔고 부족 등)과 응답을 위해 원장 전체 합계를 매번 계산하지 않아도 되게 하려는 선택입니다. 대신 변경은 항상 트랜잭션 안에서 원장/상세/잔고를 같이 갱신해 정합성을 맞춥니다.

### `POINT_DETAIL`을 둔 이유

사용/취소는 단순히 잔고만 맞추면 끝나지 않습니다. “어떤 적립 건이 어떤 주문에서 얼마 사용됐는지”를 남겨야 하고, 사용취소 시에도 같은 기준으로 복원되어야 합니다. `POINT_DETAIL`은 그 연결을 남기는 역할을 합니다.

### requestId 기반 중복요청 처리

클라이언트가 보내는 `requestId`를 기준으로 중복 요청을 방어합니다. 이미 처리된 `requestId`면 기존 거래를 재조회해서 같은 응답을 돌려줍니다. 동시 요청으로 `request_id` 유니크 충돌이 날 수 있는 경우도 있어, INSERT 실패 시 재조회로 응답을 맞추는 흐름을 같이 둡니다.

### `remain_amount`를 거래별 “남은 처리 가능 금액”으로 사용

`remain_amount`는 “적립 잔액”만 의미하지 않습니다.

- 적립(PO01/PO05): 남은 **사용 가능 잔액**
- 사용(PO02): 남은 **취소 가능 금액**

사용취소(부분 취소 포함) 처리를 위해 PO02에도 `remain_amount`를 두고, 누적 취소 이력과 함께 정합성을 검증합니다.

### 사용취소에서 만료분을 PO05로 분리한 이유

사용취소 시 원래 차감했던 적립 건이 이미 만료라면, 같은 적립 건에 직접 복원할 수 없습니다. 이 경우는 원 적립을 되살리는 대신 **PO05(재적립)**로 금액을 별도 적립해 흐름을 분리합니다. 이렇게 하면 “원 적립은 만료”라는 사실을 유지하면서도, 취소로 인한 잔고 복원은 처리할 수 있습니다.

PO05의 `request_id`는 클라이언트 키와 분리해 내부 생성 값(`SYS-{cancelRequestId}-P05-{seq}`)을 사용합니다.

### 적립취소를 미사용 적립건만 허용한 이유

이미 사용된 적립을 취소하면, 해당 사용 건과 상세 연결을 다시 풀어야 합니다. 이 프로젝트에서는 적립취소(PO03)는 “미사용 적립”만 대상으로 제한해 로직을 단순화하고 정합성 위험을 줄입니다.

## 기능별 정책

### 적립(PO01)

- 1회 적립 금액 범위와 사용자 최대 보유 금액은 설정값(`application.yml`)로 관리합니다.
- 만료일은 기본값이 있고, 입력 시에는 “최소 1일 이후 ~ 최대 5년 미만” 범위를 검증합니다.
- 관리자 수기 지급 여부는 `admin_granted_yn`으로 구분합니다.

### 적립취소(PO03)

- **PO01 / PS01 / 미사용(remain_amount == amount)** 적립만 취소 가능합니다.
- 일부라도 사용된 적립은 취소할 수 없습니다.
- 원 적립은 삭제하지 않고 `PS03` + `remain_amount = 0`으로 처리합니다.
- PO03 거래 1건을 추가하고, `POINT_DETAIL`에 PO03 1건을 남깁니다.

### 사용(PO02)

- `orderNo`는 필수입니다.
- 차감 대상 적립 건은 다음 우선순위로 선택합니다.
  1) 관리자 수기 지급(`admin_granted_yn = 'Y'`) 우선  
  2) 만료일 빠른 순  
  3) 생성 시각 순
- 사용 시 `POINT_DETAIL(PO02)`을 기록해 어떤 적립 건에서 얼마가 차감됐는지(`source_point_key`) 추적합니다.
- PO02의 `remain_amount`는 남은 취소 가능 금액으로 사용합니다(초기에는 `amount`와 동일).

### 사용취소(PO04)

- 전액/부분 취소가 가능합니다.
- 원 PO02의 `POINT_DETAIL(PO02)`을 기준으로 **FIFO**로 복원합니다.
- 복원 대상 적립 건이 만료된 경우 직접 복원하지 않고 PO05(재적립)로 대체합니다.
- PO04는 매 요청마다 `POINT_DETAIL(PO04)`를 반드시 남깁니다(부분 취소 누적과 복원 흐름 추적).
- 정합성 검증을 위해 아래 값을 맞춥니다.
  - `PO02.amount - PO02.remain_amount`
  - 원 PO02를 기준으로 집계한 PO04 누적 합계

## 테스트

- 적립/사용/적립취소/사용취소에 대한 컨트롤러 통합 테스트가 있습니다.
- 동일 사용자에 대한 동시 `use`는 정합성 확인용 스모크 테스트로 포함되어 있습니다.

## 로그 및 운영 고려사항

- 주요 흐름은 `INFO`, 상세값은 `DEBUG`로 남깁니다.
- `DEBUG`에서 `jvmMem` 로그를 남깁니다(메모리 사용량 확인용).
- 각 유스케이스(적립/적립취소/사용/사용취소)는 요청 단위로 `elapsedMs` 로그를 남깁니다.
- 중복제어는  `requestId` 기반이며, 유니크 충돌 시 재조회 패턴으로 중복 INSERT를 방어합니다.
- 동시성은 단일 인스턴스/단일 트랜잭션 기준으로 정합성을 맞추는 방식입니다. 다중 인스턴스/고동시 환경에서는 행 잠금, 원자적 UPDATE 같은 보완이 필요할 수 있습니다.
